logseq-plugin-daily-todo — research summary

Compiled 2026-05-10 · Living doc, regenerate when conclusions change

Everything we've established about how this plugin interacts with Logseq: the load-time mechanism behind the 6-second warning, the trigger semantics that govern when migration fires, the SDK breakages waiting for an SDK upgrade, the sync architecture and its multi-device implications, and the layered testing strategy. Cross-references the underlying research documents.

Overview

logseq-plugin-daily-todo is a ~300-line TypeScript plugin with two responsibilities: migrate unfinished TODO blocks from yesterday's journal into today's when Logseq creates today, and provide two keyboard shortcuts (mod+1 cycle TODO/DONE, mod+4 highlight). No UI, no network, no React. All logic lives in src/main.ts.

What follows is the durable knowledge gathered while modernizing the repo and adding test coverage — describes how the world is and why, not who did what when.

1. Load-time fix (0.0.8)

The 6-second warning

Logseq emits "this plugin takes too long to load" when a plugin's iframe-load + Postmate handshake exceeds 6000 ms. Plugins are loaded serially — a slow plugin blocks every plugin queued behind it.

  • Threshold check: src/main/frontend/handler/plugin.cljs:1276
  • Interval: libs/src/LSPlugin.core.ts:1504-1509 wraps pluginLocal.load(...). Stops timer when Postmate's handshake-reply reaches the parent.

What does & doesn't count

Counts toward the 6 s metric: iframe HTML fetch · each chained ES module import (every lsp:// fetch is an Electron IPC roundtrip) · bundle parse + execute · the Postmate handshake roundtrip itself.
Doesn't count: anything inside logseq.ready(callback), including App.registerCommandPalette, DB.onChanged, datascriptQuery, updateSettings. By the time Logseq stops the timer, those haven't run yet.

If you see the warning, don't optimize main(). Look at the bundle.

The fix shipped in 0.0.8

The old build used manualChunks: { logseq: ['@logseq/libs'] }, producing an entry chunk that did import './logseq.xxx.js' — two chained lsp:// fetches before ready() could fire. The new config sets output.inlineDynamicImports: true, producing a single ~78 KB chunk: same gzip, same code, half the per-startup IPC roundtrips.

Do not switch back to manualChunks. It looks like a vendor-cache optimization, but plugin assets aren't shared across plugins; every chunk is its own lsp:// IPC roundtrip on every plugin load on every Logseq startup.

The single-chunk invariant is enforced in pnpm perf: it drives perf-test/host.html headlessly and asserts resourceCount === 1 + a median load threshold.

2. Trigger semantics — when does migration fire

The filter is correct

The plugin's DB.onChanged filter requires createdAt === updatedAt && journal? === true on a single transaction. That shape is what Logseq's in-app create-today-journal! emits.

Don't relax this filter. Loosening it would re-fire migration on every Logseq startup as it parses existing journals, duplicating content and corrupting source journals.

Paths that DO trigger migration

  • The day-rollover timer (a 3 s setInterval in frontend.handler/watch-for-date!) firing create-today-journal! at midnight while the app is running
  • The today-link click in the sidebar on a new day
  • The fs-watcher reacting to today's file being deleted (Logseq re-runs create-today-journal!)
  • Cold start on a new day — Logseq creates today empty before the listener runs

A consequence: today's journal is always empty at the moment migration runs.

Paths that don't

  • UI navigation to today when today already exists in datascript (no creation transaction, no listener fire)
  • File-import (e.g. dropping a markdown file into the journals directory, or sync arriving from another device — see §4)
  • Any edit to an existing journal page

Why "today has pre-existing content" isn't a real bug

Earlier testing surfaced a "rule-11" case: if today already has content, migration overwrites it. Investigation showed the scenario isn't reachable through any real Logseq trigger — the file-deletion path wipes today before migration runs, and other paths don't fire the listener at all. The recursiveCopyBlocks branch at lastDestBlock.content !== '' exists for the abstract case but is dead code in practice.

Don't "clean up" the dead branch. If Logseq's create-today behavior ever changes, it could become live again.

3. SDK breakages waiting in src/main.ts

The plugin pins @logseq/libs@0.0.9. Three call sites break against any modern SDK (≥ 0.0.17):

3.1 block.left.id is gone

BlockEntity.left was removed in favor of parent: IEntityID + order: string. The empty-separator cleanup uses block.left.id; rewrite via the parent's children array, or drop the cosmetic cleanup on DB graphs.

3.2 block.content is deprecated

Read sites should use block.title ?? block.content. Write paths still accept content strings.

3.3 journal? / journalDay moved to PageEntity

Runtime values still appear in DB.onChanged payloads for journal pages — the existing dual-key check stays correct, but the static type should narrow to PageEntity.

Status: see master's refactor: prepare src/main.ts for @logseq/libs >= 0.0.17 commit. Refactor work has begun; SDK bump itself comes after.

Don't bump @logseq/libs past 0.0.x without fixing all three. The plugin will silently break.

4. Logseq Sync (paid beta) implications

What sync transmits

Whole files, not datascript transactions. Logseq Sync is a deltaful S3-style file sync delegating heavy lifting to a Rust binary (rsapi). End-to-end encrypted via age; both file contents and filenames are encrypted. Pulled files arrive on the receiving device via the fs-watcher path, NOT as fresh-creation transactions.

No primary device, no coordination

Every device runs its own watch-for-date! 3 s poll that calls create-today-journal! when db/page-empty? repo today-page. There is no midnight cron, no central coordination. db/page-empty? is a local-DB-only check.

Mobile is a passive participant

Plugins do NOT run on Logseq mobile. The Capacitor iOS/Android app gates plugin install, marketplace UI, and LSPluginCore bootstrap behind (util/electron?) checks. There is no Postmate host on mobile.

This means the plugin's migration logic only runs on desktop devices. Mobile sees the migration result through sync — it pulls today's already-migrated journal file from the server and renders it. From the user's perspective, "I opened my phone in the morning and yesterday's TODOs were already in today" is the result of migration that ran on a desktop earlier, not on the phone.

Overnight migration mechanism

A common pattern: lid is closed, the user thinks the laptop is "asleep and disconnected," yet today's journal contains yesterday's TODOs the next morning. The mechanism:

  1. macOS Power Nap (powernap = 1) wakes the system from Deep Idle every 15-30 minutes for 9-60 second DarkWake windows for routine maintenance (DHCP renewal, mail/iCloud, Spotlight, Time Machine, etc.).
  2. Logseq Electron is kept resident in RAM and was launched with --disable-features=MacWebContentsOcclusion, which skips Chromium's hidden-window throttling. During DarkWake, the renderer thread runs JavaScript.
  3. The 3 s watch-for-date! poll fires. (date/today) now returns the new day. create-today-journal! creates today's page (single tx, created-at == updated-at, journal? true).
  4. The plugin's listener matches. queryCurrentRepoRangeJournals reads from the local DataScript DB (already populated from IndexedDB cache — independent of sync). Migration runs. Markdown files are rewritten on disk.
  5. Whenever WiFi briefly comes up during a later DarkWake (tcpkeepalive=1) or on lid-open, sync uploads the modified files. Mobile pulls them next time it connects.

The migration is purely a local file-write operation that doesn't need network during DarkWake. Power Nap fires multiple times per hour with high regularity, so within ~1 hour after midnight the migration is essentially guaranteed to have run.

To verify on your machine: compare yesterday's journal mtime with pmset -g log | grep DarkWake: 

stat -f "%Sm %N" /path/to/your/graph/journals/*.md | tail -7
pmset -g log | grep DarkWake | tail -50

Mtimes for migrated files should fall inside DarkWake windows in the early hours.

Multi-device duplication is a real risk

Two scenarios produce duplicate migration:

  1. Both desktops boot offline / before sync converges. Each sees "today is empty in my local DB", each runs create-today-journal!, each fires the plugin, each runs migration locally. After sync converges, default last-writer-wins picks one; the loser lives in logseq/version-files/local/. With diff-merge enabled, page UUIDs differ → blocks concatenate → duplicates.
  2. Race on already-online desktops. Desktop A creates today's journal at 08:00:00; sync push starts. Desktop B's 3 s poll fires at 08:00:03, B's DB is still empty for today, B creates its own today's journal. Both run migration. Same outcome.

The migration is destructive on the source side (removes migrated TODOs from yesterday). If both devices migrate, both modify yesterday too, and last-writer-wins on yesterday's file means whichever lands first wins; the other device's view of yesterday gets overwritten. Some DONE/title content could be lost in the collision.

Severity: bounded probability (requires both desktops active around midnight or first-open of a new day), but real impact when it happens. Single-desktop users are not affected.

Sync ingestion does NOT re-fire migration

When sync delivers today's journal file from another device, it goes through the fs-watcher → parser path. The resulting transaction has :from-disk? true, reuses existing page entities when present, and does NOT produce a fresh-creation tx shape. The plugin's filter doesn't match. Sync delivery doesn't cause a second migration run on the receiving device.

5. Testing strategy

Three layers

LayerWhatCostIn CI
Unit (Vitest) Pure helpers extracted to src/lib.ts: regexes, getNextTodoState, group-split, decideCopyAction <1 s yes
Mocked-SDK Rejected. Mock fidelity is most of the work and has to track every SDK change. — —
End-to-end (Playwright) Drives real Logseq.app with isolated profile; asserts on on-disk markdown content. ~13 s/case, ~3 min full suite yes (Linux runners)

Verification surface in package.json: 

pnpm test             # Vitest unit tests (<1s)
pnpm perf             # Headless perf gate (~5s)
pnpm verify           # test + build + perf (umbrella)
pnpm test:e2e:quick   # mega-migration sanity (~17s)
pnpm test:e2e         # full E2E suite (~3m30s)

Headless E2E — why it works

The plugin's natural test loop ("edit → build → load unpacked into Logseq → restart → eyeball") needs a human at steps 3-4. The E2E harness replaces that with Playwright driving real Logseq.app:

  • Asserts on disk content, not API call counts. Disk is what users see, what syncs, and what survives Logseq restarts.
  • Surfaced real plugin bugs on first run — the empty-prevJournals reduce, the rule-7 DONE-child drop. Both fixed in dedicated PRs.
  • Runs in CI on Linux (e2e.yml), with the macOS-specific isolation patch skipped because Linux Electron honors HOME.

The Flow A trigger

Several trigger paths were considered:

PathResult
Click "Choose a folder" then navigate to #/page/<today>Doesn't create the journal — Logseq just shows an empty page
Call apis.doAction(['openDir']) directly Doesn't go through the create-today-journal! path
Eval the renderer's closure-private create-today-journal!Not reachable from window
Write today's journal file to disk after launch Logseq parses as content-import; tx shape doesn't match the plugin's filter
Pre-seed yesterday + today, delete today after launch Works. Logseq's fs-watcher detects, runs create-today-journal!, single-tx with createdAt === updatedAt && journal? === true — plugin's filter matches first try

Flow A mirrors a real-world manual reproduction: open Logseq, delete today's file, watch Logseq re-create it with migrated content. Same code path as the in-app today-link click and the day-rollover timer.

Shortcut testing — Escape after each press

When Logseq's editor opens on a block (via .block-content click), the block's view-mode rendering is replaced by the editor textarea. Until the editor closes, the block isn't findable via .block-content selectors. This broke chained-keystroke cases and caused state leakage between cases.

Fix: the harness sends Escape after each shortcut press to commit the edit and close the editor. Plus a defensive Escape at the start of runShortcutCase for any leftover state from a prior case.

Isolation — the macOS patch

Setting HOME=$TMP is not sufficient on macOS: Electron's app.getPath('home') calls getpwuid(getuid())->pw_dir and ignores HOME. Logseq's electron.js mixes both APIs, so an unpatched run reads/writes the user's real ~/.logseq/.

The harness hard-link-clones /Applications/Logseq.app to /tmp/LogseqPatched.app via pax -rwl (~3 s, ~0 disk because hardlinks), then breaks the hardlink only on Contents/Resources/app/electron.js and rewrites every app.getPath("home") and os.homedir() to honor LOGSEQ_FAKE_HOME. The original Logseq.app is never modified; reversible with rm -rf /tmp/LogseqPatched.app.

The runner snapshots ~/.logseq/settings/ mtimes before/after each run and fails non-zero if anything was touched.

6. AI-first repo conventions

Minimum viable kit for a 300-LOC leaf project:

  1. AGENTS.md — root front door with a decision-table for "what you're doing → which doc to read"
  2. CLAUDE.md — short redirect to AGENTS.md
  3. agents/ — topic-named guides for stable knowledge (overview, gotchas, build/release, testing, plugin-loading, perf-testing)
  4. agents/research/YYYY-MM-DD-topic.md — dated investigation logs (this doc is one)
  5. package.json scripts as the verb surface (check, test, perf, verify, test:e2e)

Topic guides answer "how does this work today." Research answers "how did we figure this out and what else did we consider." Don't mix them — mixing rots the topic guides.

Patterns deliberately skipped at this size: .claude/commands/, .claude/scripts/, agents/skills/, agents/prompts/, agents/plans/, ADRs / docs/decisions/, Cursor / Cline rules files. All overkill for a leaf project.

7. Open questions

  • Sync ingestion tx shape on a brand-new today's journal when the receiving device's DB doesn't yet have that page entity. Code-path reading suggests it's a multi-block parser tx with :from-disk? true, but worth confirming empirically.
  • Mobile sync pull timing relative to its 3 s poll. If sync usually finishes first, mobile sees today's journal already in DB after sync ingestion and skips create-today-journal!. (Mobile doesn't run the plugin anyway, so this only affects whether mobile creates duplicate today-pages.)
  • Default enable-sync-diff-merge? setting. Plugin behavior under duplicated migrations differs between last-writer-wins and three-way-merge modes.
  • DB-graph era. The user is on file graphs. DB graphs use a different sync stack (deps/db-sync/, src/main/frontend/worker/sync.cljs) that transmits datascript-level ops. Plugin behavior under that stack is a separate research question.
  • Logseq running while macOS is genuinely off. Hibernation kills the process; the migration can't run. After wake from hibernate, the next 3 s poll fires migration. Worth confirming that the user's reported "Logseq app closed" cases were actually Cmd+W (window closed, process running) vs Cmd+Q (process killed). If process killed, then autostart-on-login or the next manual app-open is when migration runs, not while genuinely powered off.
